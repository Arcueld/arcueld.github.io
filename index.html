hello-java-sec打靶
2024/11/17

sql注入

jdbc

漏洞描述

SQL注入是因为程序未能正确对用户的输入进行检查，将用户的输入以拼接的方式带入SQL语句，导致了SQL语句可控 从而获取数据库信息，造成信息泄漏

对于提供的例子 漏洞的成因都是先拼接的字符串 然后传入statement

由于错误的使用预编译 在此还是导致了sql注入

正确的做法是占位符 加set

靶场提供的安全解决方案是 黑名单 预编译 使用ESAPI框架 jdbctemplate使用强制数据类型

注意jdbctemplate也要正确使用才安全

安全编码

正确使用预编译 分离语句和数据 从本质上避免sql注入

出现异常时禁止返回堆栈 sql语句等敏感信息 统一返回错误页面 屏蔽异常栈

mybatis

使用${}表示拼接字符串 会直接将接受到参数的内容不加任何修饰符拼接在 SQL 中 从而导致sql注入

而#{}使用预编译，通过 PreparedStatement 和占位符来实现，会把参数部分用一个占位符 ? 替代

强制使用Integer或long类型 可以在一定程度上解决${}导致的sql注入 但是还是不建议用${}

@Select("select * from users where id = ${id}")
List<User> queryById2(@Param("id") Integer id);

模糊搜索时，直接使用’%#{q}%’ 会报错，部分研发图方便直接改成’%${q}%’从而造成注入

@Select("select * from users where user like '%${q}%'")
List<User> search(String q);

// 安全代码,采用concat
@Select("select * from users where user like concat('%',#{q},'%')")
List<User> search(String q);

安全编码

使用#{}进行参数绑定

针对order by注入 进行排序映射

<select id="orderBySafe" resultType="com.best.hello.entity.User">
    select * from users
    <choose>
        <when test="field == 'id'">
            order by id desc
        </when>
        <when test="field == 'user'">
            order by user desc
        </when>
        <otherwise>
            order by id desc
        </otherwise>
    </choose>
</select>

fastjson
2024/10/23

fastjson可以将javaBean转json json转javaBean

实际上就是序列化反序列化

https://javasec.org/java-vuls/FastJson.html

Demo

<dependency>
    <groupId>com.alibaba</groupId>
    <artifactId>fastjson</artifactId>
    <version>1.2.23</version>
</dependency>

通过Json.toJsonString进行序列化

parseObject进行反序列化

package com.Arcueid;

import com.alibaba.fastjson.JSON;
import com.alibaba.fastjson.serializer.SerializerFeature;


public class App 
{
    public static void main( String[] args )
    {
        Person person = new Person("Arcueid","114514");

        String jsonString = JSON.toJSONString(person, SerializerFeature.WriteClassName);
        System.out.println(jsonString);
        Person res = JSON.parseObject(jsonString,Person.class);
        System.out.println(res);
    }
}

SerializerFeature.WriteClassName属性会自动在序列化后添加@type 即写上序列化对象的类名

运行发现private的id并没有反序列化出来

这需要加上Feature.SupportNonPublicField参数

        Person res = JSON.parseObject(jsonString,Person.class, Feature.SupportNonPublicField);

parseObject 不指定类则反序列化得到的都是JSONObject对象
